# Bug Fixes - February 13, 2025

## Summary
Fixed critical issues in the registration flow that were preventing proper user onboarding and transition to workout planning.

## Issues Fixed

### 1. Weight Not Being Saved (Decimal Values)
**Problem**: User entered weight 72.5 kg, but it was saved as `null` in the database.

**Root Cause**: The `weightSchema` in `registration.validation.ts` used `.int()` constraint, which rejected decimal values.

**Solution**: 
- Removed `.int()` constraint from `weightSchema` to accept decimal values
- Added test case to verify decimal weight values (72.5 kg) are properly saved

**Files Changed**:
- `apps/server/src/domain/user/services/registration.validation.ts`
- `apps/server/tests/integration/services/registration.integration.test.ts`

---

### 2. Profile Status Transition Without Complete Data
**Problem**: `profile_status` could transition to "complete" even when required fields were missing.

**Analysis**: The validation logic was already correct - it checks that all 6 required fields are present AND user has confirmed before marking profile as complete (line 92 in `registration.service.ts`).

**Verification**: No code changes needed. The existing validation in `RegistrationService.processUserMessage()` already prevents transition without all required data.

---

### 3. No Transition to Planning Phase After Registration
**Problem**: After completing registration, the system stayed in chat phase instead of automatically transitioning to session planning phase.

**Root Cause**: In `chat.routes.ts`, the phase transition was hardcoded as `registration → chat` instead of `registration → session_planning`.

**Solution**:
- Changed phase transition from `registration → chat` to `registration → session_planning`
- Added validation for `registration → session_planning` transition in `ChatService.validatePhaseTransition()`
- Updated system note to "Registration complete. Let's plan your first workout!"

**Files Changed**:
- `apps/server/src/app/routes/chat.routes.ts` (line 84)
- `apps/server/src/domain/user/services/chat.service.ts` (added validation)

---

### 4. Double Greeting Issue
**Problem**: User received two greetings:
1. Hardcoded template from Telegram bot: "👋 Hi filko! I'm your personal fitness coach..."
2. LLM-generated greeting from the registration prompt

**Root Cause**: The `/start` command in the Telegram bot sent a hardcoded greeting, then the user's first message triggered another greeting from the LLM.

**Solution**:
- Removed hardcoded greeting from Telegram bot's `/start` handler
- Bot now sends "hi" message to LLM to get a single, personalized greeting
- LLM generates contextual greeting based on the registration prompt

**Files Changed**:
- `apps/bot/handlers.ts`

**Benefits**:
- Single, personalized greeting from LLM
- Greeting is in the user's language (Russian/English)
- Conversation starts immediately with registration flow
